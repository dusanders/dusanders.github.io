Bluetooth Low Energy RC Car

Main Components

  • Microchip RN4020 BTLE Module   |
  • TI MSP430G2553 MCU   |
  • 2x TI DRV8833 H-Bridge Drivers   |
  • 6v to 3.3v Step-down converter

Protocols Used

  • UART between MSP430 and RN4020   |
  • PWM between MSP430 and H-Bridge Drivers   |
  • Bluetooth Low Energy between Galaxy S5 and RN4020  

Overview

This project used a Samsung Galaxy S5 Sport with an Android application utilizing the accelerometer sensor as the controller.
The Android device connected with the RN4020 over Bluetooth Low Energy and sent commands over BTLE where the RN4020 would forward the commands to the MSP430 via UART at 9600 baud rate.
The MSP430 would provide logic based on the commands received and appropriately control the PWM signal send to the drivers based on what information was recieved.
The MSP430 also contained logic to stop the car if communication was lost for whatever reason.

Scrolling Display

Main Components

  • Newhaven 2x16 LCD Display Module   |
  • Renesas RL-78 MCU   |
  • Cypress 64Kbit F-RAM Module   |
  • Silicon Labs CP2104 USB-to-UART Bridge

Protocols Used

  • UART between CP2104 and RL78   |
  • I2C between RL78 and LCD Display   |
  • I2C between RL78 and F-RAM Module  

Overview

This project was completed as part of a class on embedded systems. The RL78 is programmed entirely in Assembly using IAR Embedded Workbench IDE.
The system is powered from a single AAA battery and has an external power switch for on/off operation.
The device is a simple scrolling display which reads from the internal 64Kbit F-RAM module. The device continually cycles the stored text.
The device attaches to a host machine via a USB cord, where a terminal application can deliver text to the device.
During a 'load' operation via a terminal application, the USB-to-UART bridge forwards typed characters to the RL78 which then takes those characters and writes them to the F-RAM module for future use.

'Phone Mouse'

Main Components

  • Linux 'uevent' system file   |
  • Local Network Connection   |
  • Android Application

Protocols Used

  • Linux kernel logic with C and 'uevent' file   |
  • UDP stream from Android to Linux machine   |
  • Touch events within Android  

Overview

This project allows an Android device to act as the mouse and keyboard input for a Linux machine by utilizing the 'uevent' system file within Linux.
The project creates a UDP stream across the local network between the Android device and the Linux machine. The Android device will scan the local network for available IP addresses and displays a list to the user, where they can choose the appropriate machine. The connection is performed with UDP to allow for a streaming connection, avoiding packet construction and associated overhead. The Android application captures touch events and uses a text input box for keyboard events which allows the user to view input on the Android device.

File Compression

Main Components

  • Input File (text)   |
  • Command Line   |
  • Output File (text)

Protocols Used

  • File Input/Output   |
  • Binary File Streams   |
  • 'Huffman' Algorithm  

Overview

This project was created during a Data Structures and Algorithm course. The program is a command line utility that compresses a text file using the 'Huffman' algorithm. The program uses file I/O operations to read an input file; where the text is analyzed according to the 'Huffman' algorithm and uses a rooted binary tree for storage of 'codewords'. These 'codewords' are then written to the output file within a binary stream to achieve a 57% compression ratio. Decompression is handled with file I/O and read in as a binary stream which gets deciphered and written back as the original plain text.

BTLE RC Car Controller

Main Components

  • Bluetooth Low Energy RC Car  |
  • BTLE Android Device

Technologies Used

  • BTLE Communication   |
  • Android Accelerometer I/O

Overview

This project was created to act as the controller for my BTLE RC Car. The application accesses the Android accelerometer sensor with the device and uses the tilting of the device as the input control for the RC car. Communication is performed over 5 different BTLE characteristics. The device will scan for nearby BTLE receivers and display a list to the user. Once connected, the device will access the accelerometer and begin reading the current 'tilt' of the device and send the appropriate signal to the RC car. The 'zero' reading is when the device is held flat, 'forward' recorded when the user rotates the device forward, 'backward' when the user rotates the phone backwards, and 'left'/'right' when the user rotates the devices in the respective direction.

'Phone Mouse'

Main Components

  • Android Device   |
  • Local Network Access   |
  • Linux Host Machine

Technologies Used

  • Android Touch Events   |
  • Android Keyboard Events   |
  • Android UDP Connection

Overview

This project was created in order to easily control a Linux netbook attached to my television that acts as a 'media center'. The application was produced on an older Android device (Motorola Photon) that I had laying around. The application will scan the currently connected Wifi network for available IPs where the Linux host can be selected. A UDP connection is established between the devices to allow for a 'stream' connection which is appropriate for continous input events like mouse control. The Android device captures touch event on the screen and reports them to the Linux machine as a mouse movement. A text area is available on the Android device which provides the user with a keyboard and the ability to easily read typed characters; this keyboard input is reported to the Linux machine as keyboard input.

Android Dashcam

Main Components

  • Android Device

Technologies Used

  • Android Camera Operation   |
  • Android File I/O   |
  • Android Timer Events

Overview

This project was created as a way to utilize an old Android device I had laying around. The application uses the main camera of the device and records both audio and video. The application records roughly three minutes of video; after the three minutes are recorded, the device will save the file and begin a new three minute recording. The device saves upto three files during operation, if the users presses the 'Save' button then the files are tagged as persistant, otherwise the device will overwrite the files on the next iteration. The overwrite operation was created to save memory on the device during extended periods of driving; allowing for continous recording without worry of available memory. The application also uses the accelerometer of the device to 'Save' the recordings in the event of an accident or major disruptive event.
